# Integration Patterns For Upstream TTS Runtimes

Use this file to decide how to wrap an upstream TTS without overfitting to one
project.

This reference intentionally targets practical local TTS runtimes. If the
upstream project mainly exposes notebooks, multi-stage research pipelines, or
streaming conversation primitives, it may not fit this skill cleanly.

## Pattern A: Direct Python API

Use this when the upstream project exposes a Python API that already returns
audio or writes files directly.

Typical signs:

- README shows `from <pkg> import ...`
- inference is done through a class or function call
- methods like `generate(...)`, `synthesize(...)`, or `generate_to_file(...)`
  are already documented

Preferred adapter shape:

1. Load the model once at startup or lazily on first request.
2. Map OpenAI-compatible request fields to the upstream API.
3. Prefer the upstream file-writing helper if one exists.
4. Return the generated bytes as `audio/wav`.

Examples:

- KittenTTS: `KittenTTS(...)`, `generate(...)`, `generate_to_file(...)`

## Pattern B: CLI Or Batch File Output

Use this when the upstream project already has a reliable CLI that writes a WAV
file, even if the internal Python API is less stable or less documented.

Typical signs:

- README centers commands like `./bin/piper --text ... -f output.wav`
- JSON or stdin input is already supported by the CLI
- file output is a first-class path

Preferred adapter shape:

1. Write request input into CLI arguments or stdin.
2. Generate to a temporary file.
3. Read the WAV bytes.
4. Return the bytes as `audio/wav`.

Examples:

- piper-plus: C++ CLI with `--text`, `--speaker`, `-f/--output_file`

## Pattern C: Internal Runtime Plus Save Helper

Use this when the upstream project exposes a runtime object, but raw audio
serialization is fragile unless you reuse the repo's own helper.

Typical signs:

- README or code shows a runtime object such as `InferenceRuntime`
- the project includes a `save_wav(...)` or equivalent utility
- `result.audio` type/shape is not clearly documented for external consumers

Preferred adapter shape:

1. Call the runtime directly.
2. Reuse the upstream save helper.
3. Avoid hand-rolled `soundfile` or `BytesIO` code unless the helper truly does
   not exist.

Examples:

- Irodori-TTS: `runtime.synthesize(...)` plus `save_wav(...)`

## Selection Heuristic

Choose the highest-level proven path:

1. If a stable Python API exists, start with Pattern A.
2. Else if a reliable CLI writes WAV, use Pattern B.
3. Else if an internal runtime exists with a save helper, use Pattern C.
4. Only build a custom serialization path after confirming the upstream project
   does not already solve the problem.

Reject or defer the wrapper when one of these is true:

- the upstream repo does not show a stable inference path outside notebooks
- the useful path depends on reference audio or cloning inputs that do not fit
  the default OpenAI speech request shape
- the project is really a realtime/streaming framework rather than a one-shot
  TTS engine

Practical examples:

- Good fit: Kokoro, Piper, KittenTTS, Coqui TTS in normal text-to-file mode
- Usually not the first target for this skill: StyleTTS2 research flows,
  streaming-first frameworks such as RealtimeTTS, or cloning-heavy setups that
  need extra inputs

## Field Mapping Notes

- `input`
  - Usually maps to text directly.
- `voice`
  - May map to speaker name, speaker id, or be ignored.
- `speed`
  - May map directly, inversely, or not exist at all.
- `response_format`
  - If upstream supports only WAV, accept the field but reject non-WAV values.

## Practical Warning

Do not equate "OpenAI-compatible API" with "same internal runtime contract."
The HTTP surface can be standardized while the inference path remains highly
provider-specific.
